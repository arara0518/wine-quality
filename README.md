# Wine Quality Classification

機械学習の自己学習ポートフォリオ第4弾。Kaggleのワイン品質データセットを用いて、外れ値処理・SVM/Gradient Boosting/XGBoostの比較・k-foldクロスバリデーション・3種類のハイパーパラメータ探索手法（GridSearchCV/RandomizedSearchCV/Optuna）の比較を実施した。

## 環境
- Google Colab
- Python 3.10
- scikit-learn / xgboost / optuna / pandas / numpy / matplotlib / seaborn / scipy

---

## データセット概要

| 項目 | 内容 |
|---|---|
| サンプル数 | 1,143件 |
| 特徴量 | 11個（すべて数値データ） |
| 目的変数 | quality（ワインの品質スコア、3〜8の6クラス） |
| 欠損値 | なし |

### 特徴量の説明

| 特徴量 | 意味 |
|---|---|
| fixed acidity | 固定酸度（酒石酸など揮発しにくい酸） |
| volatile acidity | 揮発酸度（酢酸など。高すぎると酢の味になる） |
| citric acid | クエン酸（新鮮さや風味に関わる） |
| residual sugar | 残糖（発酵後に残った糖分） |
| chlorides | 塩化物（塩分量） |
| free sulfur dioxide | 遊離亜硫酸（防腐剤として機能する形態） |
| total sulfur dioxide | 総亜硫酸（遊離＋結合した亜硫酸の合計） |
| density | 密度 |
| pH | 酸性度（低いほど酸性） |
| sulphates | 硫酸塩（防腐剤・抗酸化剤として添加） |
| alcohol | アルコール度数 |

## クラス不均衡の状況

qualityの分布は深刻な不均衡を示した。

| quality | 件数 | 割合 |
|---|---|---|
| 3 | 6 | 0.5% |
| 4 | 33 | 2.9% |
| 5 | 483 | 42.3% |
| 6 | 462 | 40.4% |
| 7 | 143 | 12.5% |
| 8 | 16 | 1.4% |

quality=5・6（中間品質）に82.7%が集中し、極端な品質（3・4・8）は合計でも4.8%しかない。この不均衡が分類精度全体、特に少数クラスの予測に大きく影響した。

## 外れ値処理

箱ひげ図でほぼ全列に上方の外れ値が確認された。residual sugar（残糖）・chlorides（塩化物）・sulphates（硫酸塩）などで特に多くの外れ値が見られた。

### 処理方針：Winsorizing（上下1%）
California Housingと同様、データ件数を保ったまま極端な値を抑制するためWinsorizingを採用した。処理後も1,143件のまま維持された。

## モデル比較

スケーリングはSVM用にStandardScalerを適用し、Gradient Boosting・XGBoostにはスケーリングなしの元データを使用した。

### 結果（train/test split 1回評価）

| モデル | Accuracy | Macro F1 | Weighted F1 |
|---|---|---|---|
| SVM (rbf kernel) | 0.6769 | 0.32 | 0.65 |
| Gradient Boosting (sklearn) | 0.6638 | 0.33 | 0.65 |
| XGBoost | 0.6812 | 0.40 | 0.67 |

XGBoostが全指標で最良だった。特にmacro F1が0.40と他の2モデルより明確に高く、quality=8を3モデル中唯一予測できた（precision 0.33・recall 0.33）点が大きな違いだった。

### Gradient BoostingとXGBoostの違い
両者は同じ勾配ブースティングの考え方（弱い決定木を直列に積み上げ、前の木の誤差を次の木が修正する）に基づくが、XGBoostはL1/L2正則化が組み込まれている点、欠損値を自動で扱える点、二次近似を使った高精度な分割基準を持つ点で異なる。今回XGBoostが優れた結果を出した一因として、正則化機構が少数クラスへの過学習を抑えつつ精緻な分割基準でその特徴を捉えられた可能性が考えられる。

### クラス不均衡が3モデル共通で与えた影響
quality=3・4は3モデルとも一度も正解できなかった（サンプル数が極端に少ないため）。quality=5・6（多数派クラス）はどのモデルも安定して高いrecallを示したが、quality=7・8では精度が大きく低下した。

## k-foldクロスバリデーション

StratifiedKFold（5分割）でXGBoostの性能を再評価した。層化（stratify）を用いた理由は、quality=3が6件しかない状況で通常のk-foldを使うとfoldごとにクラス分布が大きく偏る可能性があるためである。

結果は次の通りである。各foldのaccuracyは0.5983、0.6026、0.6550、0.6272、0.6886であり、平均accuracyは0.6343、標準偏差は0.0339だった。

### train/test split（0.6812）と5-fold CV平均（0.6343）の差についての考察
1回のtrain/test split評価は5-fold CVの平均より高い値（0.6812）を示したが、これは「たまたま当てやすい分割を引いていた」可能性が高い。5-foldの各スコアには0.5983〜0.6886という約9%の幅があり、評価が分割方法に大きく依存することを直接示している。これは少数クラス（quality=3・4・8）を含むデータセットで典型的に見られる現象であり、k-fold CVの平均（0.6343）の方が単発のtrain/test split評価より信頼できる性能推定値と判断した。

## ハイパーパラメータ探索手法の比較

GridSearchCV・RandomizedSearchCV・Optunaの3手法でXGBoostのハイパーパラメータ最適化を試みた。いずれもscoring='accuracy'で5-fold CVを用いて評価した。

### 結果

| 手法 | Accuracy | 試行数 | 計算コスト |
|---|---|---|---|
| デフォルト（5-fold CV） | 0.6343 | - | - |
| GridSearchCV | 0.5932 | 27通り全探索 | 高 |
| RandomizedSearchCV | 0.5941 | 2400通り中30通りサンプリング | 中 |
| Optuna | 0.5993 | ベイズ最適化30回 | 中 |

### 探索手法間の効率性についての考察
3手法すべてがデフォルトパラメータを下回ったが、探索手法間では効率の差が確認できた。GridSearchCVは全探索（27通り）にもかかわらず最も低い結果となり、探索範囲自体が狭すぎた可能性がある。RandomizedSearchCVはGridSearchCVよりはるかに広い探索空間（2400通り）からわずか30回の試行でほぼ同等の結果を達成した。Optunaは同じ30回の試行でRandomizedSearchCVをわずかに上回り（0.5993 対 0.5941）、過去の試行結果を活用するベイズ最適化の効率性が現れた結果と考えられる。

### 3手法すべてがデフォルトに勝てなかった理由についての考察
考えられる理由は以下の3点である。scoring='accuracy'を最大化する設計が、少数クラス（quality=3・4・8）を無視し多数派クラスへの適合を優先する方向にパラメータを調整してしまった可能性。少数クラスへの配慮はaccuracyという指標自体に組み込まれていないため、これは構造的な問題である。XGBoostのデフォルトパラメータが、この程度の規模・クラス不均衡を持つデータセットに対してすでに十分頑健に設計されている可能性。探索空間自体に「真に良いパラメータ」の組み合わせが含まれていなかった可能性。

この結果は「ハイパーパラメータチューニングは必ずしも精度改善をもたらすわけではない」という実務的な教訓を示している。少数クラスの存在を考慮するなら、次回はscoring='f1_macro'など不均衡に強い指標で再探索することが改善の方向性として考えられる。

## 全体を通した考察

このデータセットの核心的な課題はクラス不均衡である。quality=3が6件、quality=8が16件という極端な少数派を統計的に学習することは構造的に困難であり、どのモデル・どのハイパーパラメータ探索を使っても限界がある。XGBoostが相対的に最良の結果を示したが、これはアルゴリズムの優劣だけでなく、正則化による少数クラスへの過学習抑制が効いた可能性が大きい。

今後この課題に対処するには、Phase 3で扱うSMOTE（少数クラスの合成サンプル生成）やclass_weightの調整など、不均衡データに特化した手法が必要になる。
